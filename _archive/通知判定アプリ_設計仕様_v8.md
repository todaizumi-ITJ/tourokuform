# 通知判定アプリ 設計仕様書 v8

更新日: 2026-03-05

---

## アーキテクチャ概要

Flask + BeautifulSoup4 + openpyxl による新規案件通知判定Webアプリ。
新規案件HTMLファイルを受け取り、既存DBと照合して弁護士への通知要否を自動判定する。

---

## データファイル構成（最小完全系）

### db_human.json（32,742件）
既存案件DB。STEP1の氏名照合に使用。

```json
{
  "hn_id": "HN22232K",
  "contract_name": "山田太郎",
  "contract_name_kana": "やまだたろう",
  "address": "東京都新宿区...",
  "mandate_status": "受任済",
  "cp_nums": [19],
  "rights_holder": "株式会社WILL",
  "settlement_status": "終了",
  "kmp_progress": "終了",
  "lawyer_name": "春田 藤麿",
  "lawyer_office": "春田法律事務所"
}
```

### db_ipn.json（16,853件）
IPN → CP・CW対応表。新規HTMLのIPNからCP・CWを解決する唯一のキー。
全件IPNあり（100%）。

```json
{
  "095FFF": { "cp_num": 19, "cw_num": 7859, "rh": "株式会社WILL" }
}
```

### db_settled.json（22件）
包括和解済みDB。hn_id × cp_num の組み合わせで管理。

```json
[
  { "hn_id": "HN2226JP", "cp_num": 3, "stl_settled_at": "2023-04-03" },
  { "hn_id": "HN222WPE", "cp_num": 8, "stl_settled_at": "2025-12-23" }
]
```

**旧settled_comprehensive.jsonから不要フィールドを除去した軽量版。**
- `stl_type` / `stl_status` は不要（このファイルにある時点で包括和解済みを意味する）
- `rights_holder` 文字列は不要（cp_numがあればcp_masterから引ける）

## 通知判定ロジック（v8）

```
新規案件 1件 入力
        │
        ▼
【前提】IPNなし → ❌ 通知しない（CP判定不能）
        │
        ▼
【IPN解決】db_ipn.json で ipn_id → cp_num / cw_num / rh を取得
        │
        ├─ IPN未解決（DBに存在しない）→ ❌ 通知しない（安全側）
        │
        └─ 解決成功
                │
                ▼
        【STEP1】氏名正規化 → db_human.json と照合
                │
                ├─ マッチなし → ✅ 通知する（完全新規）
                │
                └─ マッチあり
                        │
                        ▼
                【STEP3】条件①: 受任済み × 同一CP
                        条件: mandate_status='受任済' かつ cp_nums に cp_num が含まれる
                        │
                        ├─ 該当 → ❌ 通知しない（弁護士が既に対応中）
                        │
                        └─ 非該当
                                │
                                ▼
                        【STEP4】条件②: 包括和解済み（hn_id × cp_num）
                                条件: (マッチしたhn_id, 新規cp_num) が db_settled に存在
                                │
                                ├─ 該当 → ❌ 通知しない（包括和解済み）
                                │
                                └─ 非該当 → ✅ 通知する
```

### v7からの変更点

| 項目 | v7（旧） | v8（新） |
|------|---------|---------|
| 条件②の判定単位 | cp_numのみ（全HN共通） | hn_id × cp_num の組み合わせ |
| settled管理 | cpセット `{1,2,3,4,5,6,8}` | HN×CP組み合わせ 22件 |
| 誤抑制リスク | 同姓同名の別人HNでも停止 | 実際に和解したHNのみ停止 |

## 正規化ルール

### 氏名
1. Unicode NFKC正規化（全角英数→半角）
2. 空白・全角スペース除去
3. 異体字統一（斎→斉、邉→辺、髙→高、﨑→崎、嶋→島、濱→浜 など）

### 住所
1. 全角数字→半角
2. 丁目/番地/号→ハイフン統一
3. 前40文字で正規化、照合は前20文字（確実一致）/ 前12文字（高確度）

## 判定ランク

| ランク | 条件 | 意味 |
|--------|------|------|
| ✅ 通知する（新規） | 氏名マッチなし | 完全新規 |
| ✅ 通知する（既存） | 条件①②非該当 | 既存あり・対応不要と判断 |
| ❌ 条件① | 受任済み×同一CP | 弁護士対応中 |
| ❌ 条件② | 包括和解済み（HN×CP） | 対象HNが対象CPと和解済み |
| ❌ IPNなし | IPNなし | CP判定不能 |
| ❌ IPN未解決 | IPNあるがDB未登録 | 安全側に倒す |
