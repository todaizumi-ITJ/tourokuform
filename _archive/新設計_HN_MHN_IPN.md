# 新設計書：HN・MHN・IPN 統合アーキテクチャ

作成日: 2026-03-05

---

## 1. 設計思想

発信者情報開示請求によってIPと人が紐づく。この流れを軸に設計する。

```
権利者がIPを把握（IPN発行）
        ↓
プロバイダに発信者情報開示請求
        ↓
名前・住所・メールが判明
        ↓
IPN + 人情報 がセットで届く（新規案件HTML）
```

IPNは著作物・権利者・人を結びつける核心情報。IPNがあれば判定に必要な情報はすべて揃う。

---

## 2. HNの必須フィールド定義

### パターンA：IPN付き（主流）

| フィールド | 必須/任意 | 理由 |
|-----------|---------|------|
| IPN | **必須** | 著作物・CP・CWがすべて解決できる |
| 氏名 | **必須** | 人物特定 |
| 住所 | **必須** | 人物特定 |
| IP | 任意 | IPNで代替可能 |
| タイムスタンプ | 任意 | IPNで代替可能 |
| プロバイダ | 任意 | 記録として推奨 |
| 権利者 | 任意 | IPNから解決できる |

### パターンB：IPNなし（少数）

| フィールド | 必須/任意 | 理由 |
|-----------|---------|------|
| IPN | — | なし |
| IP | **必須** | IP+タイムスタンプでIPN後引き可能 |
| タイムスタンプ | **必須** | IP+タイムスタンプでIPN後引き可能 |
| プロバイダ | **必須** | 発信者情報開示請求の根拠として必要 |
| 氏名 | **必須** | 人物特定 |
| 住所 | **必須** | 人物特定 |
| 権利者 | 任意 | あれば判定精度が上がる |

## 3. IPN解決フロー

```
新規案件 入力
        │
        ├─ IPNあり
        │      ↓
        │   db_ipn.json で解決
        │      ├─ 解決成功 → cp_num / cw_num / rh 取得 → 判定へ
        │      └─ 未解決   → IP+タイムスタンプで再検索（後述）
        │
        └─ IPNなし
               ↓
           IP + タイムスタンプ で db_ip_lookup.json を検索
               ├─ IPN発見（90%想定）→ 通常判定に合流
               ├─ IPN未発見・新規   → ✅ 通知する（安全側）
               └─ IPN未発見・既存あり → ⏸ 判定保留（目視確認）
```

IP+タイムスタンプによるIPN後引きで、IPNなし案件の約90%は通常判定に合流できる。

## 4. MHN（マスターHN）設計

### 定義

MHNは「同一人物を束ねる親ID」。

```
MHN-001（人Aのマスター）
  ├─ HN22232K（新DB）← D#1234（旧DB、CP3）
  ├─ HN22345X（同一人物の別CP案件）← D#1235（旧DB、CP5）
  └─ D#1236（CP3・包括和解後に追加）
```

### HNに持たせるフィールド

```json
{
  "hn_id": "HN22232K",
  "mhn_id": "MHN-001",
  "mhn_status": "confirmed",
  ...
}
```

### mhn_status の3状態

| 状態 | 意味 | 通知判定 |
|------|------|---------|
| `confirmed` | 同一人物確定・MHN付与済み | 通常判定を実行 |
| `unknown` | 照合不明・目視確認待ち | 暫定で✅通知する（安全側） |
| `new` | 完全新規・新MHN発行済み | ✅通知する |

**unknownでも通知判定を止めない。** 不明のまま通知する方向に倒し、目視確認後にMHNを付与して再判定する。

---

## 5. MHN付与の自動判定ルール

| 照合結果 | mhn_status | 処理 |
|---------|-----------|------|
| 氏名一致 ＋ 住所前20文字一致 | `confirmed` | 自動でMHN付与 |
| 氏名一致 ＋ 住所前12文字一致 | `confirmed` | 自動でMHN付与 |
| 氏名一致 ＋ 住所不一致 | `unknown` | 目視確認リストへ |
| カナのみ一致 | `unknown` | 目視確認リストへ |
| マッチなし | `new` | 新MHN発行・通知する |

---

## 6. D# との関係

D#（旧DB）は「人 × CP」単位で生成されていた。HN（新DB）は「人」単位。

```
【同一人×同一CPで複数D#があるケース】
D#1234（CP3・個別和解済み）
D#1236（CP3・包括和解に移行）
    ↓
どちらもMHN-001にぶら下がる
判定には「より新しい和解状況（包括和解）」を優先する
```

### D# と IPN の紐づき

- D# と IPN は紐づいている（1:1とは限らない）
- IPNからD#を引くことでCP・CW・和解状況が特定できる
- IPNなし案件はD#との紐づけをIP+タイムスタンプで代替する

## 7. 通知判定フロー（新設計統合版）

```
新規案件 入力
        │
【STEP0】IPN解決
        IPNあり → db_ipn.json
        IPNなし → IP+ts → db_ip_lookup.json
        いずれも未解決 → mhn_status='unknown' → ⏸ 暫定通知する
        │
【STEP1】氏名・住所照合 → MHN判定
        confirmed / unknown / new を付与
        │
        ├─ new（マッチなし）→ ✅ 通知する（完全新規）
        │
        └─ confirmed / unknown
                │
        【STEP2】条件①: 受任済み × 同一CP
                │
                ├─ 該当 → ❌ 通知しない
                └─ 非該当
                        │
                【STEP3】条件②: 包括和解済み（MHN × CP）
                        │
                        ├─ 該当 → ❌ 通知しない
                        └─ 非該当 → ✅ 通知する
```

---

## 8. 判定結果の3分類

| 分類 | 意味 |
|------|------|
| ✅ 通知する | 弁護士に通知が必要 |
| ❌ 通知しない | 受任済み or 包括和解済み |
| ⏸ 判定保留 | IPN未解決・目視確認が必要（暫定で通知する扱い） |

---

## 9. 和解後ダウンロードチェック（別プロセス）

IPN検出タイムスタンプが和解成立日より後の場合は別案件として扱う。

```
IPN検出日 > 和解成立日（stl_settled_at）
    ↓
⚠️ 和解後ダウンロード → 別レポートに出力
    ├─ hn_id / mhn_id / contract_name
    ├─ cp_num / rights_holder
    ├─ ipn_id / cw_num / title
    ├─ 和解成立日 / IPN検出日 / 経過日数
    └─ 弁護士確認フラグ
```
