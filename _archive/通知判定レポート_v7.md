## v7の変更点（ロジックの大幅簡素化）

### 旧ロジック（v5〜v6）の問題
IPNがない場合にrights_holderの文字列で補完しようとしていたが、
表記ゆれ・既存HNの権利者空欄・1対多マッピング等の問題でCP判定が不正確だった。

### 新ロジック（v7）
**IPNなし → 一律通知しない。** CP判定は cp_num（数値）のみで行う。

```
新規案件 1件
  ↓
IPNあり？
  ├─ なし → ❌ 通知しない（IPN未付与・CP判定不能）
  └─ あり → IPN → cp_num 解決
               ↓
         mandate DBに存在するか？
           ├─ なし → ❌ 通知しない（未知のIPN）
           └─ あり → cp_num取得
                       ↓
               STEP1: 氏名照合
                 ├─ マッチなし → ✅ 通知する（完全新規）
                 └─ マッチあり
                       ↓
                   STEP2: 受任済み × 同一CP（cp_num数値一致）
                     ├─ 該当 → ❌ 通知しない（条件①）
                     └─ 非該当
                           ↓
                       STEP3: 包括和解済みCP（settled_cpと数値一致）
                         ├─ 該当 → ❌ 通知しない（条件②）
                         └─ 非該当 → ✅ 通知する
```

## 技術メモ

### v7の簡素化ポイント
- RH補完ロジック（rh_cp_map）を完全廃止
- cp_numの数値比較のみで条件①②を判定
- rights_holder文字列照合をロジックから除去
- 表記ゆれ・空欄問題が構造的に発生しない

### 根本的な解決策
IPNが全件付与されれば v7 のロジックで完全に正確な判定が可能。
現状はIPNあり89件（12%）のみ判定精度が保証される。
IPNなし657件（88%）は安全側として通知しない扱い。

### 使用データ
- db_human.json（32,742件）: 既存案件DB
- db_ipn.json（16,853件）: IPN→cp_num/rights_holder対応表
- db_settled.json: 包括和解済みCP（cp_nums: [1,2,3,4,5,6,8]）
