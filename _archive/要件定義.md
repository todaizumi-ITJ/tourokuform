# 通知判定システム 要件定義書

作成日: 2026-03-05
バージョン: 1.0

---

## 1. システム概要

新規案件データ（HTMLスプレッドシート形式）が届いたとき、既存DBと照合し弁護士への通知要否を自動判定するシステム。

---

## 2. データ構造と用語定義

### 2.1 旧DB（アダムス）との関係

| 旧DB用語 | 新DB用語 | 説明 |
|---------|---------|------|
| D# | HN | 案件ID。旧DBでは「人 × 権利者（CP）」ごとに生成 |
| - | cp_num | 権利者（CP）の数値ID |
| IPN | IPN / ipn_serial | 著作物識別子。D#に紐づく（1:1とは限らない） |

### 2.2 D# の生成ルール（旧DB）

- D#は「1人 × 1権利者（CP）」の組み合わせごとに生成される
- 同一人が複数のCPと案件を持つ場合、CP数分のD#が存在する
- 同一人 × 同一CP でも、以下のケースで複数のD#が存在しうる：
  - 個別和解を先に行い、その後に包括和解に移行した場合
  - この場合、旧DBに同一人×同一CPで複数のD#レコードが存在する

### 2.3 D# と IPN の紐づき

- D# と IPN は紐づいている
- 1対1とは限らない（1D# に複数IPNが存在する可能性あり）
- IPNから逆引きすることで CW（著作物番号）と CP（権利者）が特定できる

## 4. 通知判定ロジック（現行 v8）

```
新規案件 1件 入力
        │
【前提】IPNなし → ❌ 通知しない（CP判定不能）
        │
【IPN解決】db_ipn.json で ipn_id → cp_num / cw_num / rh を取得
        │
        ├─ IPN未解決 → ❌ 通知しない（安全側）
        │
        └─ 解決成功
                │
        【STEP1】氏名正規化 → db_human.json と照合
                │
                ├─ マッチなし → ✅ 通知する（完全新規）
                │
                └─ マッチあり
                        │
                【STEP3】条件①: 受任済み × 同一CP
                        mandate_status='受任済' かつ cp_nums に cp_num が含まれる
                        │
                        ├─ 該当 → ❌ 通知しない（弁護士が既に対応中）
                        │
                        └─ 非該当
                                │
                        【STEP4】条件②: 包括和解済み（hn_id × cp_num）
                                マッチしたhn_id と 新規cp_num の組み合わせが
                                db_settled に存在するか
                                │
                                ├─ 該当 → ❌ 通知しない（包括和解済み）
                                │
                                └─ 非該当 → ✅ 通知する
```

## 5. 包括和解の定義と判定

### 5.1 包括和解と個別和解の違い

| 種別 | 内容 |
|------|------|
| 個別和解 | 特定の著作物（CW）単位での和解 |
| 包括和解 | 特定の権利者（CP）との全著作物を対象とした和解 |

### 5.2 旧DBにおけるフラグ

旧DBには和解日・契約締結日の記録がない。以下のフラグで包括和解の可能性を推定する：

| フラグ状態 | 判定 |
|-----------|------|
| `stl_type = 'comprehensive'` かつ `stl_status = '終了'` | 包括和解確定 |
| 合意後入金中 | 包括和解の可能性高 |
| 分割入金中 | 包括和解の可能性高 |
| 終了フラグのみ | 個別和解完了の可能性あり（要確認） |

### 5.3 複数D#問題

同一人 × 同一CPで以下のケースが存在しうる：

```
人A × CPx の場合
├─ D#-1: 個別和解で終了
└─ D#-2: 包括和解（後から移行）
    → 新DBでは同一HN × 同一cp_numとして扱われる
    → 現行判定（hn_id × cp_num）でカバーされているが、
      個別和解か包括和解かの区別は現状できない
```

## 6. 和解後ダウンロードチェック（別プロセス・未実装）

### 6.1 問題の定義

IPNのタイムスタンプ（違法DL検出日時）が、既存HNの和解成立日より**後**である場合、和解後に別の著作物を違法ダウンロードしたことになる。これは弁護士確認が必要な別案件として扱う。

### 6.2 判定フロー

```
インプット：
  ・新規IPNの検出タイムスタンプ（ipn_timestamp）
  ・マッチしたHNの和解成立日（stl_settled_at）

判定：
  IPN検出日 > 和解成立日 → ⚠️ 和解後ダウンロード（別プロセスへ）
  IPN検出日 ≤ 和解成立日 → 通常判定フロー続行
```

### 6.3 出力レポート（別シート・別ファイル）

| フィールド | 内容 |
|-----------|------|
| hn_id | 案件ID |
| contract_name | 氏名 |
| cp_num / rights_holder | 権利者 |
| ipn_id / cw_num / title | 著作物情報 |
| stl_settled_at | 和解成立日 |
| ipn_timestamp | IPN検出日 |
| 経過日数 | 検出日 − 和解日（日数） |

## 7. 正規化ルール

### 氏名

1. Unicode NFKC正規化（全角英数→半角）
2. 空白・全角スペース除去
3. 異体字統一（斎→斉、邉→辺、髙→高、﨑→崎、嶋→島、濱→浜、瀧→滝、廣→広、澤→沢、眞→真 など）

### 住所

1. 全角数字→半角
2. 丁目/番地/号→ハイフン統一
3. 前40文字で正規化し、照合は前20文字（確実一致）/ 前12文字（高確度）で比較

## 9. 判定ランク

| ランク | 条件 | 意味 |
|--------|------|------|
| ✅ 通知する（新規） | 氏名マッチなし | 完全新規 |
| ✅ 通知する（既存） | 条件①②非該当 | 既存あり・通知必要 |
| ❌ 条件① | 受任済み×同一CP | 弁護士対応中 |
| ❌ 条件② | 包括和解済み（HN×CP） | 対象HNが対象CPと和解済み |
| ❌ IPNなし | IPNなし | CP判定不能・安全側 |
| ❌ IPN未解決 | IPNあるがDB未登録 | 安全側 |

## 10. 現状の課題

| 課題 | 内容 | 影響 |
|------|------|------|
| IPN付与率が低い | 新規HTML全体の9.2%のみIPNあり | 90%が「IPNなし→通知しない」で判定不可 |
| IPN未解決 | 87件のIPNがdb_ipn.jsonに存在しない | 安全側に倒してすべて通知しないとなる |
| cp_nums 不完全 | 未受任HNはcp_numsが空のケースあり | 条件①の精度に影響 |
| 和解後DL未検出 | 和解後のタイムスタンプ照合がない | 和解後の再違法DLを検出できない |
| 包括和解種別なし | db_settled.jsonにstl_typeがない | 個別/包括の区別ができない |
| 手動更新 | DB更新はすべて手動 | 定期更新がない |
