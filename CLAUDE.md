# 通知判定システム 仕様書

更新日: 2026-03-06

---

## 1. システムの目的

著作権侵害案件について、弁護士への通知が必要かどうかを自動判定する。

---

## 2. データの種類と入口

届くデータは2種類あり、**入口の時点で完全に分離する**。

### 種類A：プロバイダ開示通知（通知判定の対象）

- 送付元：プロバイダ（ソフトバンク、au等）
- 形式：HTML / Excel
- 特徴：`lawyer_id` が**空**
- 処理：通知判定フローへ

### 種類B：弁護士受任通知（通知判定の対象外）

- 送付元：弁護士事務所
- 形式：Excel / CSV
- 特徴：`lawyer_id` が**入っている**
- 処理：弁護士案件DBへ登録のみ。通知判定しない。

> **原則：弁護士がすでに動いている案件は通知不要。入口で弾く。**

---

## 3. 通知判定フロー（種類Aのみ）

```
プロバイダ開示通知 1件
        │
【前提チェック】
        │
        ├─ lawyer_id あり → ❌ 通知しない（弁護士案件・誤混入）
        │
        ├─ IPN なし      → ❌ 通知しない（CP判定不能）
        │
        └─ IPN あり
                │
        【IPN解決】db_ipn.json で ipn_id → cp_num / cw_num / rh 取得
                │
                ├─ 解決失敗（DB未登録）→ ❌ 通知しない（安全側）
                │
                └─ 解決成功
                        │
                【STEP1】氏名正規化 → db_human.json と照合
                        │
                        ├─ マッチなし → ✅ 通知する（完全新規）
                        │
                        └─ マッチあり
                                │
                        【STEP2】条件①：弁護士対応中チェック
                                マッチした既存案件が以下のいずれかかつ同一CP
                                ・mandate_status = '受任済'
                                ・mandate_status = '受任確認中'
                                │
                                ├─ 該当 → ❌ 通知しない（弁護士が対応中）
                                │
                                └─ 非該当
                                        │
                                【STEP3】条件②：包括和解済みチェック
                                        マッチした (hn_id × cp_num) が
                                        db_settled.json に存在する
                                        │
                                        ├─ 該当 → ❌ 通知しない（包括和解済み）
                                        │
                                        └─ 非該当 → ✅ 通知する
```

---

## 4. 通知しない条件まとめ

| 条件 | 判定理由 |
|------|---------|
| lawyer_id あり | 弁護士案件（入口で除外） |
| IPN なし | CP判定不能 |
| IPN 未解決（DB未登録） | 安全側に倒す |
| 受任済 または 受任確認中 × 同一CP | 弁護士が既に対応中 |
| 包括和解済み（hn_id × cp_num） | 対象HNが対象CPと既に和解済み |

## 5. データファイル構成

| ファイル | 件数 | 役割 |
|---------|------|------|
| db_human.json | 32,742件 | 既存案件DB（STEP1照合用） |
| db_ipn.json | 16,853件 | IPN → {cp_num, cw_num, rh} 対応表 |
| db_settled.json | 22件 | 包括和解済み（hn_id × cp_num） |
| db_ip_lookup.json | 16,840件 | IP+タイムスタンプ → IPN検索用 |

### db_human.json の主要フィールド

```json
{
  "hn_id": "HN22232K",
  "contract_name": "山田太郎",
  "address": "東京都新宿区...",
  "mandate_status": "受任済",        // 未受任 / 受任済 / 受任確認中
  "cp_nums": [19],
  "settlement_status": "終了",
  "lawyer_name": "春田 藤麿",
  "lawyer_office": "春田法律事務所"
}
```

### db_ipn.json の構造

```json
{ "095FFF": { "cp_num": 19, "cw_num": 7859, "rh": "株式会社WILL" } }
```

### db_settled.json の構造

```json
[{ "hn_id": "HN2226JP", "cp_num": 3, "stl_settled_at": "2023-04-03" }]
```

## 6. 正規化ルール

### 氏名
1. Unicode NFKC正規化（全角英数→半角）
2. 空白除去
3. 異体字統一（斎→斉、邉→辺、髙→高、﨑→崎、嶋→島、濱→浜、瀧→滝、廣→広、澤→沢、眞→真 等）

### 住所
1. 全角数字→半角
2. 丁目/番地/号→ハイフン統一
3. 照合は前20文字（確実一致）/ 前12文字（高確度）

## 8. 出力（Excel）シート構成

| シート | 内容 |
|-------|------|
| 📋 全件判定結果 | 全案件の判定結果 |
| ✅ 通知する | 通知対象のみ |
| ❌ 通知しない | 抑制対象のみ |
| ❌ 通知しない（詳細） | 同姓同名候補を全件展開 |

---

## 9. 実装（app.py）

- バックエンド：Python / Flask
- HTMLパース：BeautifulSoup4
- Excel出力：openpyxl
- DB形式：JSON（ハッシュインデックスで高速化）
- 処理速度：1,000件 × 32,742件を約0.2秒

### 主なエンドポイント

| エンドポイント | 機能 |
|-------------|------|
| `POST /api/judge` | HTMLファイル → 通知判定 |
| `POST /api/export` | 判定結果 → xlsx出力 |
| `POST /api/lookup` | IP+タイムスタンプ → IPN検索 |
| `POST /api/reload-db` | DBをホットリロード |
| `GET /api/db-status` | DB件数確認 |

## 10. 現状の課題

| 課題 | 影響 |
|------|------|
| IPN付与率が低い（9.2%のみ） | 90%がIPNなし→通知しない扱い |
| IPN未解決（698件） | db_ipn.jsonに未登録のIPN（ナチュ・エムズ・MBM・ベンハー等） |
| mandate_status='受任確認中'がSTEP2に未追加（要修正） | 受任確認中が通知されてしまう |
| 包括和解の日付（stl_settled_at）活用未実装 | 和解後ダウンロードを検出できない |
